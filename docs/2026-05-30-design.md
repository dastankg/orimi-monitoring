# Минимальный учебный стек мониторинга: Prometheus + Grafana + Loki

**Дата:** 2026-05-30
**Статус:** одобрено к реализации
**Охват:** только сервис `orimi` (Django-бэкенд)

## Цель

Поднять минимальный, но рабочий observability-стек для изучения Prometheus,
Grafana и Loki. Метрики приложения (django-prometheus) и контейнеров (cAdvisor)
в Prometheus, логи всех контейнеров в Loki, единый UI — Grafana.

Не цель: алертинг, продакшн-grade HA, мониторинг ботов (добавим позже тем же
шаблоном).

## Архитектура

> Расположение: стек живёт отдельной папкой `orimi-monitoring/` рядом с
> сервисами (у каждого сервиса свой git-репозиторий — это не монорепо).
> Связь с orimi — через общую docker-сеть `monitoring`, а не через git.

```
┌──────────── orimi-monitoring/docker-compose.yml ────────────────────┐
│  Prometheus ──scrape──▶ web:8000/metrics   (django-prometheus)       │
│      │       ──scrape──▶ cadvisor:8080      (CPU/RAM/сеть)            │
│      ▼                                                                │
│  Grafana ◀── Prometheus (datasource)                                 │
│      ▲                                                                │
│      └─── Loki ◀── Promtail (docker-логи всех контейнеров)            │
└──────────────────────────────────────────────────────────────────────┘
   общая external docker-сеть `monitoring`
   наружу на хост (127.0.0.1): только Grafana :3000 (+ Prometheus :9090 для учёбы)
```

Сборщик логов — **Promtail** (минимальный конфиг; deprecated в пользу Alloy, но
для изучения берём как наиболее распространённый в гайдах).

## Компоненты стека (новые файлы)

Всё в папке `orimi-monitoring/`:

| Файл | Назначение |
|---|---|
| `docker-compose.yml` | 5 сервисов: prometheus, grafana, loki, promtail, cadvisor |
| `prometheus/prometheus.yml` | scrape-цели: `imagebot-web:8000`, `cadvisor:8080` |
| `loki/loki-config.yml` | single-binary, хранение на диске (filesystem) |
| `promtail/promtail-config.yml` | docker service discovery → Loki |
| `grafana/provisioning/datasources/datasources.yml` | авто-подключение Prometheus + Loki |

Тома: `prometheus_data`, `grafana_data`, `loki_data`.

## Изменения в сервисе `orimi` (Django)

| Файл | Изменение |
|---|---|
| `requirements.txt` | `+ django-prometheus` |
| `configs/settings.py` | `django_prometheus` в `INSTALLED_APPS`; `PrometheusBeforeMiddleware` первым и `PrometheusAfterMiddleware` последним в `MIDDLEWARE`; `SECURE_REDIRECT_EXEMPT = [r"^metrics$"]` |
| `configs/urls.py` | `path("", include("django_prometheus.urls"))` → endpoint `/metrics` |
| `gunicorn.conf.py` (новый) | хук `child_exit` → `multiprocess.mark_process_dead(worker.pid)` |
| `entrypoint.sh` | очистка+создание `PROMETHEUS_MULTIPROC_DIR`; gunicorn с `-c gunicorn.conf.py` |
| `nginx.conf` | `location /metrics { deny all; }` — закрыть публичный доступ |
| `docker-compose.yml` (orimi) | `web` в сети `default` + `monitoring`; env `PROMETHEUS_MULTIPROC_DIR`; объявить external-сеть `monitoring` |

## Ключевые технические решения

1. **Multiprocess-метрики при 4 воркерах gunicorn.** Без этого `/metrics`
   отражает случайного воркера на каждый скрейп → счётчики «прыгают». Решение:
   `PROMETHEUS_MULTIPROC_DIR` + `child_exit`-хук. django-prometheus при наличии
   этой переменной автоматически агрегирует через `MultiProcessCollector`.
   Каталог чистится при старте (entrypoint), т.к. stale-файлы воркеров искажают
   данные.

2. **SSL-редирект ломает скрейп.** В settings включён `SECURE_SSL_REDIRECT`.
   Prometheus ходит по HTTP на `web:8000`, получил бы 301. Исключаем `/metrics`
   через `SECURE_REDIRECT_EXEMPT`.

3. **Безопасность `/metrics`.** Endpoint открыт без аутентификации, поэтому
   публично закрываем его в nginx (`deny all`). Prometheus скрейпит контейнер
   `web:8000` напрямую внутри docker-сети, минуя nginx.

4. **Кросс-compose сеть.** Общая external-сеть `monitoring`, создаётся вручную
   (`docker network create monitoring`). К ней подключены `web` (из orimi) и
   сервисы мониторинга. cAdvisor и Promtail читают docker-сокет хоста.

## Порядок реализации

1. Создать external-сеть `monitoring`.
2. Правки приложения orimi (requirements, settings, urls, gunicorn, entrypoint,
   nginx, compose).
3. Файлы стека мониторинга (compose + конфиги + provisioning).
4. Проверка: `docker compose config` обоих файлов; пересборка web; открыть
   Grafana, убедиться что метрики и логи приходят.

## Проверка результата

- `web:8000/metrics` отдаёт метрики (изнутри сети `monitoring`).
- В Grafana → Explore: PromQL `rate(django_http_requests_total_by_method_total[5m])`
  возвращает данные.
- В Grafana → Explore (Loki): `{container="imagebot-web"}` показывает логи.
- cAdvisor-метрики (`container_memory_usage_bytes`) доступны в Prometheus.
